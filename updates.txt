i3d Blender importer — update log
==================================

2026-05-13  Multi-axis joint rotations: Euler convention fix
------------------------------------------------------------

Symptom:
  Skeletons imported with the wrong orientation/position for FS25 animals whose
  joints have rotations with more than one non-zero component — pigSow, sheep,
  horse, waterBuffalo. The skeleton appeared offset from the body (e.g. floating
  above and behind the mesh). Cow, goat, and lamb were unaffected because their
  joint rotations were single-axis (e.g. spine_01 = (0, -90, 0)).

Root cause:
  The importer used Blender Euler mode 'ZYX' to interpret GIANTS' intrinsic
  ZY'X'' rotations, on the assumption that the names matched. They don't.
  Empirically, Blender's `Euler((rx, ry, rz), 'ZYX').to_matrix()` builds
  M = Rx @ Ry @ Rz (extrinsic ZYX / intrinsic XYZ), NOT M = Rz @ Ry @ Rx
  (intrinsic ZYX, which is what GIANTS uses).

  Blender Euler order names read right-to-left in matrix multiplication, so:
    'ZYX' → Rx @ Ry @ Rz   (wrong for GIANTS)
    'XYZ' → Rz @ Ry @ Rx   (matches GIANTS intrinsic ZY'X'')

  Single-axis rotations are identical under both conventions, which is why the
  bug went unnoticed on most rigs.

Fix:
  Changed Euler order from 'ZYX' to 'XYZ' wherever Blender Euler objects
  interpret GIANTS-side rotations:
    - io_import_i3d/importer/armature_builder.py: _local_matrix
    - io_import_i3d/importer/scene_builder.py: _make_empty_for_node + the
      axis-conversion baking block
    - io_import_i3d/importer/anim_apply.py: keyframe rest / pose interpretation
  Documented the convention reasoning in axis_convert.py module docstring.

Verified on pigSow: spine_01 now lands inside the body's back; head/jaw/ear
bones now follow the snout/forehead instead of floating behind the rear.
Same fix applies automatically to sheep, horse, waterBuffalo.


2026-05-13  v10 .i3d.shapes support
-----------------------------------

Symptom:
  Some FS25 animals' main meshes never appeared on import; only the cube empties
  for Shape nodes remained. Affected files: horse.i3d, goat.i3d, sheepEwe.i3d,
  pigSow.i3d, waterBuffalo.i3d (and posed variants).

Root cause:
  shapes_reader.SUPPORTED_VERSIONS was range(2, 10) — rejecting version 10. The
  exception ("Unsupported .i3d.shapes version 10") was swallowed by the
  top-level try/except in import_i3d.run, which just logged a warning and left
  the Shape nodes as empties. Probing v10 files against v9 layout logic showed
  identical framing — IDs, names, vertex/corner counts, options, attachments
  all decode cleanly.

Fix:
  Extended SUPPORTED_VERSIONS to range(2, 11) in
  io_import_i3d/importer/shapes_reader.py. No layout changes needed.


2026-05-13  Preserve source joint TRS on each bone (round-trip fidelity)
------------------------------------------------------------------------

Why:
  Blender bones store orientation indirectly via head/tail/roll. The importer
  computes the tail heuristically (single child → child.head; multi-child →
  best-aligned-along-+X; leaf → head + dir_x * bone_display_size). That gives a
  visually reasonable bone but the resulting bone matrix doesn't equal the
  source joint matrix exactly — any exporter reading the rotation back from
  Blender will produce ZYX Eulers that differ from the original i3d, and
  animations referencing those bones will play in the wrong pose space.

Change:
  After bone creation, each Blender Bone (in armature data) now carries the
  source i3d values as custom properties:
    _i3d_node_id
    _i3d_translation       (3-tuple of floats)
    _i3d_rotation_zyx_deg  (3-tuple of degrees, intrinsic ZY'X'')
    _i3d_scale             (3-tuple of floats)
  These survive save/reload. An exporter can write them straight back without
  depending on Blender's synthesized tail/roll.
  See io_import_i3d/importer/armature_builder.py near build_armature's end.


2026-07-02  Animated models now round-trip: graft the skeleton from the anim i3d
----------------------------------------------------------------------------

Symptom:
  Re-exporting a skinned FS25 animal and putting it back in GIANTS Editor with
  its stock animation made the mesh distort / "explode" / flail all over the
  place the moment the animation played. Adding the missing root node let the
  clip *run* but it was still broken. Restoring node transforms with the remap
  tool (remap_node_ids.py --restore-transforms) made it explode even at rest.

Root causes (there were THREE, stacked):

  1. Wrong id space. FS keeps the animation in a SEPARATE skeleton-only i3d
     (e.g. cattleCalfAnimations.i3d) that ends in <Animation externalAnimFile>.
     animals.xml marries it to a model via `animation="..."` + skeletonIndex.
     The .i3d.anim addresses bones by nodeId IN THE ANIMATION FILE's numbering
     (calf: calf_root=4, spine=5 ... eyeball_R=20), NOT the model's
     (cattleCalfHolstein: calf_root=20 ...). Feeding fix-i3d.bat the Holstein
     model remapped bones to Holstein ids, so the anim's "id 20" track
     (eyeball_R) landed on calf_root, etc. — every bone driven by the wrong
     channel. Confirmed by decoding the .anim: BoneTrack.bone_node_id values
     are 4..48 (the animation file's ids).

  2. Wrong rest pose. .i3d.anim keyframes are ABSOLUTE local transforms (they
     SET each bone every frame, not deltas). GIANTS builds the skin bind from
     the model's rest pose. A Blender round-trip mangles bone rest ORIENTATIONS
     (spine_01 exports as 90 -0 180 instead of the original 0 -90 0), so the
     bind pose disagrees with the frame the anim drives -> distortion.

  3. Why the bind can't be patched afterward. Restoring node transforms in the
     exported .i3d fixes the rest but desyncs the skin bind (vertices were bound
     to Blender's frame) -> explodes even at rest. Bind frame and anim frame
     must agree, and that is decided at import, not export.

Why the Blender round-trip can't preserve the skeleton (the hard tradeoff):
  Blender always draws a bone along its own frame's +Y, and StjerneIdioten's
  i3dio exports a bone-to-bone rest as a PURE `parent.matrix_local^-1 @
  self.matrix_local` (no bone-axis correction — verified in
  node_classes/skinned_mesh.py). So setting each EditBone's matrix to the exact
  joint frame WOULD make i3dio re-emit the original rest pose... but then the
  bones point sideways (along each joint's +Y), not down the chain. You cannot
  have both "bones follow the path" and "exported rest matches the joint" — no
  uniform correction C can decouple them (C would have to commute with every
  transform => C = ±I). We tried exact-orientation bones; the sideways look was
  rejected and the change reverted. armature_builder.py stays on geometric
  point-at-child bones.

Fix (the one that works) — graft the skeleton from the ANIMATION i3d:
  New tool io_import_i3d/tools/graft_skeleton.py + graft-i3d.bat drag-drop.
  The skeleton still goes through Blender so you can weight the mesh, but we do
  NOT ship the mangled copy. The graft:
    - discards the export's Blender skeleton,
    - drops in the skeleton straight from the animation i3d (correct rest pose
      AND the exact nodeIds the .anim references), with --preserve-skel-ids,
    - re-points every mesh's skinBindNodeIds to the grafted bones BY BONE NAME
      (canonicalised: strips Blender .001 dup-suffix and NN_ order prefix),
    - renumbers any colliding export mesh ids automatically,
    - adds <Animation externalAnimFile=...> so GIANTS Editor loads the clip.
  Result: correct rest + correct ids + your edited mesh => the stock .i3d.anim
  plays correctly. CONFIRMED WORKING on cattleCalfHolstein (all 16 skinned
  shapes + animation). This is the canonical path for MESH-ONLY edits and
  supersedes remap_node_ids.py --restore-transforms for animated models.

  Usage:
    graft-i3d.bat <exported.i3d>          (picker asks for the animation i3d)
    python io_import_i3d/tools/graft_skeleton.py <animation.i3d> <export.i3d> \
        <output.i3d> --preserve-skel-ids --anim-ref <name>.i3d.anim

Gotchas learned:
  * The graft only keeps what's in the export — ALL mesh parts must be present
    before grafting. A body-only export (wrong-name export, or not exporting the
    eyes/teeth/ears/hair objects) yields a body-only result. The calf should
    have 16 skinBindNodeIds lists; verify before shipping.
  * The collision proxy comes out of Blender WITHOUT its collision flags
    (kinematic / density / collisionFilterGroup / nonRenderable). Restore those
    from the stock proxy before shipping the mod.
  * .i3d.anim header has a 4-byte ALIGN after the character-name string, before
    clip_count (calf = 41 clips). anim_reader.parse_anim currently misses this
    align and mis-reads clip_count — fix when resuming the in-Blender anim work.

  A plain graft (in-game, where binding is structural via skeletonIndex + names
  and model ids don't matter) is also supported by sourcing the skeleton from
  the stock model and omitting --preserve-skel-ids.


2026-07-05  In-Blender animation now works (cattle): full .i3d.anim decode + apply
-----------------------------------------------------------------------------

Summary:
  Animation import INTO Blender — previously experimental/broken — now plays
  correctly for cattle (calf, highland, and adult/angus). All clips in every
  cattle animation file import (200+ across the 5 files); the rig deforms the
  way it does in GIANTS Editor. Turn on "Import animations" and (optionally)
  point "Animation i3d" at the animations file, or leave it blank to auto-detect.

The .i3d.anim format is fully decoded:
  * Each keyframe is (translation.xyz, Euler-XYZ-radians). GIANTS' intrinsic
    ZY'X'' == Blender Euler order 'XYZ' (same convention fix as the 2026-05-13
    skeleton work).
  * File header has a 4-byte ALIGN after the character-name string before
    clip_count — without it clip_count read as garbage and only clip 0 parsed.
    Fixed: all 41 cattle clips parse.
  * Track order is ALPHABETICAL by bone name. Track 0 is header-less
    ("short_intro"); it's the alphabetically-first ANIMATED bone (cow_belly for
    cattle), resolved by elimination (min of the bones not covered by an
    explicit-id track). An earlier version sent it to the root, moving the whole
    rig "off by a node".
  * Sparse clips (the 2-frame runFwd*) keyframe fewer bones than the clip
    header's bone_count; anim_reader now stops the track list at its real end
    via _looks_like_track_header (reserved==0 + sane id/kf_count) instead of
    reading into the next clip's name and desyncing the file.

Apply is forward-kinematic on the DEFORMATION (see anim_apply.py docstring):
  D[b] = src_world @ JA[b] @ JR[b]^-1 @ src_world^-1, target[b] = D[b] @ matloc.
  Rest FK (JR) is rebuilt from each bone's stored _i3d_translation /
  _i3d_rotation_zyx_deg. Bones are imported with use_connect=False so keyframe
  TRANSLATION animates (a connected bone locks its head, killing e.g. the
  tongue's extension). Normal point-at-child bones deform correctly — no
  "exact orientation" (sideways) bones needed.

Two fixes landed today that made the ADULT/Angus clips correct:

  1. Time-based sampling for sparse per-bone keyframing (the adult fix).
     Highland/calf clips are UNIFORM — every animated bone shares the same
     keyframe times (e.g. 35), so sampling all bones by index i lines up. ADULT
     clips are SPARSE: within one clip different bones have different keyframe
     counts AND times (walkFwdL: some bones 35, others 34/33/28, eight bones a
     single keyframe). The old shared-timeline-by-index apply then paired each
     bone's frame-i with the WRONG time — e.g. a bone whose 48 keyframes sit in
     0.7-4.8s got index-clamped to its 9s end-pose and froze for the whole
     middle of the clip. Fix: build a timeline from the UNION of all bones'
     keyframe times and sample every bone BY TIME (bisect to its own bracketing
     keyframes, lerp translation + slerp rotation). Highland/calf output is
     unchanged (uniform times => exact hits, no interpolation).

  2. Per-track drop instead of whole-clip skip.
     The only tracks the decoder still mis-reads are the belly (short_intro)
     track of the 2 adult runFwd* clips (1 of 45 tracks each — a sparse-layout
     oddity). anim_apply now drops just that one track (the bone holds rest) and
     keeps the clip, so runFwdL/R load and play with their other 44 bones,
     instead of being skipped entirely. Every cattle clip now loads.

Bone mapping was already correct: the .anim addresses bones by nodeId in the
ANIMATION i3d's numbering; import_i3d builds anim_id_to_name from that file and
resolves each track by name. Auto-detect follows <Animation externalAnimFile>
to an EXISTING .anim and skips round-trip outputs (_fixed/_GE/_export), so it
picks the right animations i3d even when many breeds share one differently-named
file (every adult breed uses cattleAdultAnimations.i3d.anim).

Files: importer/anim_reader.py, importer/anim_apply.py, importer/armature_builder.py
(use_connect=False, stores source T/R/S), importer/import_i3d.py (auto-detect).

Status / scope: confirmed on cattle (calf, highland, adult/angus). Other animals
(sheep, pig, horse, ...) are untested for in-Blender playback but the decode and
the sparse-sampling fix are format-level, so most should work; the short_intro
bone resolution and the separate-anim-file marriage are the parts most likely to
need a per-species check. The stock-anim round-trip via graft-i3d.bat remains a
separate, already-solved workflow for getting a re-exported model to play in GE.


--- Notes for future debugging --------------------------------------------

When troubleshooting bone alignment:
  * Confirm via Blender Python console:
      import bpy
      arm = bpy.data.objects['<armature_name>']
      print(arm.matrix_world)
      bone = arm.data.bones['<bone_name>']
      print(bone.head_local, bone.tail_local)
    head_local is the absolute armature-local rest position — bypasses pose
    mode and any UI quirks.
  * arm_obj.matrix_world is set to identity in build_armature; the bone heads
    carry the full world transform (baked from arm_root_empty.matrix_world).
    This decouples the alignment from Blender's parent/parent_inverse machinery
    which can otherwise be stale relative to scene_builder's late rotation
    updates.
  * If a bone position disagrees with hand-computed XML walk math, suspect the
    Euler convention before anything else.

Installation gotcha:
  The repo's live io_import_i3d/ folder is NOT automatically the running
  Blender addon. When testing fixes, rebuild io_import_i3d.zip and reinstall
  via Edit > Preferences > Add-ons > Install from disk. Blender extracts a copy
  to %APPDATA%\Blender Foundation\Blender\<ver>\scripts\addons\io_import_i3d
  and uses that copy. Deleting that folder when removing the addon avoids stale
  code leaking into the next install.
